# Getting started

A literal, copy-pasteable walkthrough: install the engine, create a data repo,
create a ticket, and move it across the board. Every command below is real —
run them in order and you'll have a working board in a couple of minutes.

## Prerequisites

- Node 20+
- `git` on `PATH`

## Install

```bash
npm i -g @hjr15/blaze-board
# or, without installing anything globally:
npx @hjr15/blaze-board <command>
```

The rest of this guide uses the bare `blaze` form. If you're using `npx`,
prefix every command with `npx @hjr15/blaze-board`.

## Step 1 — create a data repo

Blaze's engine (the CLI and board you just installed) holds no tickets of its
own. Tickets live in a separate **data repo** — just a directory with its own
git history.

```bash
mkdir my-tracker && cd my-tracker
git init
```

## Step 2 — tell the engine where the board lives

```bash
mkdir -p projects/ENG
cat > blaze.config.json <<'EOF'
{ "key": "ENG", "projects": ["ENG"] }
EOF
git add -A && git commit -m "init board"
```

There's no `init` subcommand — the engine finds a board by looking for a
`projects/` directory it can discover (current directory, or
`BLAZE_PROJECTS_DIR`). `mkdir -p projects/ENG` and the config file above are
what make this directory discoverable as a board with one project, `ENG`.

## Step 3 — create your first ticket

```bash
blaze new --project ENG --type task "Fix the export bug" --estimate 30
```

`task`, `story`, and `bug` all require `--estimate` (minutes, rounded to the
nearest 5) — there's no such thing as an unestimated unit of delivery work in
Blaze. The command allocates the next id (`ENG-1`), writes a schema-valid
ticket file into `projects/ENG/defined/`, and commits it.

You should see a new commit and a new file. Every ticket `new` creates gets
the same scaffolded body — `## Context`, `## Acceptance Criteria` with one
empty checkbox, and `## Notes` — for you or your agent to fill in.

A representative minimal ticket file looks like this:

```markdown
---
id: ENG-1
title: Fix the export bug
type: task
project: ENG
priority: medium
resolution:
parent:
assignee: unassigned
labels: []
components: []
estimate: 30
created: 2026-07-18
updated: 2026-07-18
---

## Context

## Acceptance Criteria

- [ ]

## Notes
```

That's the whole ticket: frontmatter plus a markdown body, sitting under
`projects/ENG/defined/` — `defined` because that's the initial status for the
delivery workflow (epic/story/task/bug/subtask). Nothing here is generated by
a database; you could hand-write this file yourself and it would be equally
valid.

## Step 4 — open the board

```bash
blaze board   # → http://localhost:4321
```

Open `http://localhost:4321`. You'll see `ENG-1` sitting in the `defined`
column of the **Board** view, with **List**, **Metrics**, and **Live** views
switchable from the header. Click the ticket id to open the detail panel —
the rendered description, full frontmatter, and (once you add them)
parent/children and links.

## Step 5 — move it forward

```bash
blaze move ENG-1 in-progress
```

This is the whole model in one command: `move` validates that
`defined → in-progress` is a legal transition, then physically relocates the
file —

```
projects/ENG/defined/ENG-1-fix-the-export-bug.md
  → projects/ENG/in-progress/ENG-1-fix-the-export-bug.md
```

— and commits the move. Refresh the board and `ENG-1` has jumped columns,
because the column is nothing but a directory listing.

Log time against it as you work:

```bash
blaze log ENG-1 45
```

This appends a worklog entry (minutes rounded to the nearest 1) to the ticket
file and commits.

## Step 6 — close it (optional)

Move it through the rest of the delivery workflow:

```bash
blaze move ENG-1 in-review
blaze move ENG-1 done
```

Entering the terminal status (`done`) auto-sets `resolution: done` — you
don't set it by hand. If a ticket needs to close for a different reason
without physically moving it through the workflow, use `resolve` instead:

```bash
blaze resolve ENG-1 wont-do   # done | wont-do | duplicate | cannot-reproduce
```

## Every command is a commit

By default (`commitMode: per-op`), every mutating command above committed
immediately, scoped to only the files it touched — `blaze new` is one commit,
each `blaze move` is one commit, `blaze log` is one commit. There's a `batch`
mode that queues operations and flushes them with one `blaze commit`
instead — useful when an AI agent is driving many operations in a session.
See [Driving Blaze with an AI agent](driving-with-an-ai-agent.md).
